# CML-140 — JSON_TO_MAPPA_DATI_ADAPTER_DESIGN_AUDIT

## Obiettivo

Progettare (non implementare) un adapter JavaScript che, dato un file JSON normalizzato, generi una struttura dati compatibile con `renderMappaDisciplinare()` (formato `*_MAPPA_DATI`). L'adapter deve consentire al runtime di consumare i JSON come fonte dati canonica, eliminando la dipendenza da dati hardcoded.

## Vincoli

- Non modificare runtime, JSON, validatore, schema `.cml`, CSS, JS applicativo
- L'adapter è un progetto — mai implementato in questa slice

## Passo 2 — Firma dell'adapter

```javascript
/**
 * Genera struttura MAPPA_DATI da JSON normalizzato.
 * @param {Object} normalizedJson - Oggetto JSON completo (root + unitaApprendimento + sezioni aggregate)
 * @returns {Object} Struttura compatibile con renderMappaDisciplinare()
 *   { disciplina: string,
 *     struttureSostanziali: Array<{id,nome,descrizione,fonte}>,
 *     nodiDisciplinari?: Array<{id,etichetta,tipo,descrizione,fonte,legami}>,
 *     progressioneVerticale: Array<{id,ordine,fascia?,classe?,descrizioneProgressione,nodiRiferimento?,fonte}>,
 *     decisioniCurricolari?: Array<{id,tipo,nodiRiferimento,unitaRiferimento,motivazione,stato,fonte}>
 *   }
 */
function toMappaDati(normalizedJson) { ... }
```

### Criteri di firma

- **Input**: un singolo oggetto JSON normalizzato (lettura del file già avvenuta)
- **Output**: oggetto compatibile con `renderMappaDisciplinare()` (campi letti: `disciplina`, `struttureSostanziali`, `nodiDisciplinari`, `progressioneVerticale`, `decisioniCurricolari`)
- **Puro**: nessun side effect, nessuna dipendenza dal DOM
- **Testabile**: output deterministico dato lo stesso input

## Passo 3 — Aggregazione ambito → struttureSostanziali

### Logica

1. Leggere `normalizedJson.unitaApprendimento`
2. Raggruppare unità per `ambito` (stringa esatta)
3. Per ogni gruppo, generare un elemento `struttureSostanziali`:
   - `id`: `stru_{disc}_{idx+1}` (es. `stru_geo_001`)
   - `nome`: il valore di `ambito`
   - `descrizione`: sintesi breve dal contenuto delle unità (prima competenza o testo statico)
   - `fonte`: `"Derivato da ambito '{ambito}' — unità {id1}, {id2}, ..."`

### Esempio

```javascript
// Input: unità con ambito "Osservazione e analisi di oggetti, strumenti e ambienti"
// Output:
{ id: "stru_tec_001",
  nome: "Osservazione e analisi di oggetti, strumenti e ambienti",
  descrizione: "Esplorazione sensoriale e funzionale di oggetti tecnologici",
  fonte: "Derivato da ambito 'Osservazione e analisi di oggetti, strumenti e ambienti' — unità tec_inf_3_001, tec_pri_1_001" }
```

### Problema: `descrizione`

Le MAPPA_DATI hardcoded hanno `descrizione` scritta a mano, non derivabile automaticamente. Soluzioni:

- **A**: usare prima competenza del gruppo come descrizione
- **B**: usare stringa vuota e lasciare che l'utente la compili
- **C**: usare il campo `ambito` come descrizione (ridondante ma sicuro)

**Raccomandato**: **A** — prendere la prima `competenza` del gruppo, è informativa e sempre presente.

## Passo 4 — Aggregazione nucleo → nodiDisciplinari

### Logica

1. Leggere `normalizedJson.unitaApprendimento`
2. Raggruppare unità per `nucleo` (o `nucleoFondante`, se presente)
3. Per ogni gruppo, generare:
   - `id`: `nod_{disc}_{idx+1}`
   - `etichetta`: il valore del nucleo
   - `tipo`: `"sostanziale"` (fisso)
   - `descrizione`: prima competenza del gruppo o testo descrittivo
   - `fonte`: `"Derivato da nucleo '{nucleo}' in {id1}, {id2}, ..."`
   - `legami`: `[]` (array vuoto, come nelle MAPPA_DATI esistenti)

### Problema: solo 3/7 JSON hanno `nodiDisciplinari` già aggregati

Tecnologia, Italiano e Matematica li hanno già nel JSON. Le altre 4 discipline (Scienze, Storia, Geografia, Inglese) hanno solo `nucleo` nell'unità. L'adapter deve **sempre** generare `nodiDisciplinari` per aggregazione, anche se già presenti nel JSON — per coerenza.

### Gestione `nucleo` vs `nucleoFondante`

- Se l'unità ha `nucleoFondante` non vuoto → usare quello
- Altrimenti → usare `nucleo`
- Tecnologia ha `nucleoFondante: ""` (vuoto) — cade nel fallback `nucleo` che è anche vuoto → si raggruppa per `ambito`. Questo è un'anomalia da risolvere nel modello dati di Tecnologia.

**Raccomandato**: normalizzare Tecnologia a usare `nucleo` come tutte le altre discipline (o popolare `nucleoFondante`), ma questo esula dall'adapter — è un fix del modello dati.

## Passo 5 — Aggregazione progressioneVerticale

### Logica

1. Leggere `normalizedJson.unitaApprendimento`
2. Raggruppare unità per `{ordine, classe || fascia}`
3. Per ogni gruppo, generare:
   - `id`: `pve_{disc}_{idx+1}`
   - `ordine`: dal gruppo
   - `fascia`: se Infanzia, altrimenti `null`
   - `classe`: se Primaria/Secondaria, altrimenti `null`
   - `descrizioneProgressione`: prima competenza del gruppo (o descrizione sintetica)
   - `nodiRiferimento`: array di id dei nodi coinvolti (riferimento a `nodiDisciplinari[].etichetta` o direttamente `nucleo`)
   - `fonte`: `"Derivato da unità {id1}, {id2}, ..."`

### Problema: `nodiRiferimento`

Le MAPPA_DATI hardcoded hanno `nodiRiferimento: []` vuoti per Tecnologia, mentre Matematica e Italiano li popolano con riferimenti a `nod_*_001`. La logica di collegamento è: se una progressione riguarda unità con un certo nucleo, il `nodiRiferimento` punta al `nodoDisciplinare` corrispondente.

**Raccomandato**: popolare `nodiRiferimento` con gli id dei `nodiDisciplinari` i cui `etichetta` corrispondono ai `nucleo` delle unità aggregate.

## Passo 6 — Gestione decisioniCurricolari

### Stato attuale

- 3/7 JSON hanno `decisioniCurricolari` (Tecnologia, Italiano, Matematica)
- Le MAPPA_DATI hardcoded le includono esattamente come nel JSON
- Non c'è corrispondenza 1:1 con dati unit — sono dati sintetici creati manualmente

### Strategia adapter

- Se il JSON ha `decisioniCurricolari` → pass-through diretto (già compatibile)
- Se il JSON non ha `decisioniCurricolari` → non generare (omettere dal output)
  - `renderMappaDisciplinare()` controlla `if (dati.decisioniCurricolari && ...)` — sezione non mostrata se assente
- Non tentare di generare decisioniCurricolari — richiedono intervento umano

## Passo 7 — Adapter per TECNOLOGIA_NORM (1:1)

Il mapping per `TECNOLOGIA_NORM` è diretto:

- `TECNOLOGIA_NORM.unitaApprendimento[i]` = `normalizedJson.unitaApprendimento[i]`
- I campi sono identici (id, disciplina, ordine, classe, fascia, ambito, competenza, traguardo, obiettivi, conoscenze, abilita, evidenze, criteriValutazione, fonte, stato, validazioneUmana, noteDipartimento)

**Adapter secondario**:

```javascript
function toTecnologiaNorm(normalizedJson) {
  return {
    schemaVersion: 'cml-normalized-curriculum-v1',
    disciplina: normalizedJson.disciplina,
    humanValidationRequired: normalizedJson.humanValidationRequired,
    source: normalizedJson.source,
    unitaApprendimento: normalizedJson.unitaApprendimento, // 1:1
  }
}
```

**Nota**: il nome `TECNOLOGIA_NORM` è fuorviante — non è specifico di Tecnologia. Dovrebbe essere generalizzato a `*_NORM` per ogni disciplina.

## Passo 8 — Gestione discipline senza sezioni aggregate

Scienze, Storia, Geografia, Inglese (4/7) hanno:

- `unitaApprendimento` (completo) ✅
- Nessuna sezione aggregata nel JSON ❌

L'adapter può generare tutte le sezioni aggregate **esclusivamente** dalle unità:

- `struttureSostanziali` ← da `ambito` (presente in tutte le unità)
- `nodiDisciplinari` ← da `nucleo` (presente in tutte le unità)
- `progressioneVerticale` ← da `{ordine, classe, fascia}` (presente in tutte le unità)
- `decisioniCurricolari` ← non generabile (sezione assente nell'output)

**Conseguenza**: 4/7 discipline avranno mappa completa tranne `decisioniCurricolari`, che resterà vuota finché qualcuno non la compilerà manualmente nel JSON.

## Schema riassuntivo adapter

```
normalizedJson
├── disciplina ──────────────────────────→ MAPPA_DATI.disciplina  [1:1]
├── unitaApprendimento[] ─────────────────→ NORM.unitaApprendimento [1:1]
│   ├── ambito ───────────────────────────→ struttureSostanziali[].nome [raggruppa per ambito]
│   │   └── + prima competenza ───────────→ struttureSostanziali[].descrizione
│   ├── nucleo / nucleoFondante ──────────→ nodiDisciplinari[].etichetta [raggruppa per nucleo]
│   │   └── + prima competenza ───────────→ nodiDisciplinari[].descrizione
│   └── {ordine, classe, fascia} ─────────→ progressioneVerticale[] [raggruppa per ordine+classe]
│       └── + prima competenza ───────────→ progressioneVerticale[].descrizioneProgressione
├── struttureSostanziali[] (se presenti) ─→ [IGNORATI, rigenerati sempre]
├── nodiDisciplinari[] (se presenti) ─────→ [IGNORATI, rigenerati sempre]
├── progressioneVerticale[] (se presenti) ─→ [IGNORATI, rigenerata sempre]
└── decisioniCurricolari[] (se presenti) ─→ MAPPA_DATI.decisioniCurricolari [pass-through]
```
