## Scelta: Opzione A — UDA read-only da unità Tecnologia

### Motivazione

1. **Pattern già rodato**: stesso approccio read-only di CML-068 (Valutazione/Evidenze), nessuna sorpresa
2. **Dati già sufficienti**: 13 unità Tecnologia hanno obiettivi, conoscenze, abilità, evidenze, criteri — base solida per una scheda UDA
3. **Basso rischio**: nessun editor, nessuna persistenza, nessun export — esattamente come il prototipo valutazione
4. **Valore alto**: il docente vede il collegamento diretto tra curricolo normalizzato e progettazione didattica
5. **Confine pulito**: Didattica > Progettazione (come da contratto CML-065)
6. **Progressione naturale**: evidenze (CML-068) → UDA (CML-070) → attività (futuro)
7. **Rischio istituzionale zero**: microcopy "modello didattico non approvato", "da adattare alla classe"

### Perimetro CML-070 — UDA_READONLY_PROTOTYPE

**Cosa include:**

- Una sezione/tab "UDA" dentro l'area Didattica (affiancata a Valutazione/Evidenze)
- 1–2 schede UDA modello costruite da unità Tecnologia normalizzata
- Campi proposti: titolo, disciplina, ordine/classe, durata indicativa, competenza, traguardo, obiettivi, conoscenze, abilità, attività proposta, metodologia, materiali, evidenze, criteri, adattamenti, nota docente, fonte, validazione umana
- Layout read-only, card espandibile, filtro ordine
- Microcopy chiara: "modello didattico non approvato", "da adattare alla classe"
- Dati embedded (nessuna fetch esterna)

**Cosa NON include:**

- Editor / campi modificabili
- Salvataggio (localStorage/sessionStorage)
- Export nuovi file
- Modifica schema `.cml`
- Generazione automatica contenuti
- Backend/API/login/OAuth
- Modifiche a Curriculum o role-access gating
