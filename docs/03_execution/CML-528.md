## Analisi install

Stato corrente dopo CML-527:

- `caches.open(CACHE_NAME)` apre la cache attiva;
- ogni `cache.add(url)` ha `catch` locale;
- un asset mancante non blocca piu l'intero `install`;
- `self.skipWaiting()` forza l'attivazione del nuovo service worker.

Rischio residuo: se `caches.open` fallisce per quota, policy browser o storage non disponibile, l'install puo ancora fallire. Questo e accettabile come rischio infrastrutturale, ma il contratto offline dovrebbe distinguere chiaramente errore cache globale da asset opzionale non recuperabile.

## Analisi activate

Stato corrente:

- cancella cache con nome diverso da `CACHE_NAME`;
- `Promise.all(keys.map(...))` riceve `undefined` per la cache corrente, ma `Promise.all` accetta valori non-Promise;
- `self.clients.claim()` prende controllo dei client.

Non sono emersi percorsi `respondWith(undefined)` in `activate`, perche `activate` non usa `respondWith`.

## Analisi fetch

### Ramo non-GET

```js
if (request.method !== "GET") return;
```

Nessun `respondWith()` viene chiamato. Il browser gestisce la richiesta con il comportamento di rete standard. Non e un caso `respondWith(undefined)`.

### Ramo navigation / HTML

```js
fetch(request)
  .then(response => response)
  .catch(() => caches.match(request).then(cached => cached || caches.match("./index.html")))
```

Percorso a rischio:

1. rete fallisce;
2. `caches.match(request)` non trova nulla;
3. `caches.match("./index.html")` non trova nulla per cache vuota, install parziale, quota, cancellazione storage o primo avvio interrotto;
4. la Promise passata a `event.respondWith()` si risolve a `undefined`.

### Ramo GET non navigation

```js
caches.match(request).then(cached => {
  if (cached) return cached;
  return fetch(request).then(...).catch(() => caches.match("./index.html"));
})
```

Percorso a rischio:

1. cache miss;
2. rete fallisce;
3. `caches.match("./index.html")` non trova nulla;
4. la Promise passata a `event.respondWith()` si risolve a `undefined`.

Rischio aggiuntivo: per asset non HTML, il fallback a `index.html` puo restituire HTML al posto di CSS, JSON, immagine o altra risorsa. Il microfix successivo dovrebbe separare navigazioni HTML da asset/statici e restituire una `Response` controllata per i casi offline non risolvibili.

## Contratto offline proposto

1. Le navigazioni HTML devono tentare rete-first, poi cache della richiesta, poi `./index.html`, poi una `Response` HTML controllata con status 503 se nessuna cache e disponibile.
2. Gli asset same-origin devono tentare cache-first, poi rete, poi una `Response` controllata con status 504 o 503 se non disponibili.
3. Le richieste cross-origin non devono ricadere su `index.html`; se falliscono, devono produrre una `Response` controllata o lasciare fallire la rete in modo intenzionale.
4. `APP_SHELL` deve contenere solo risorse essenziali per primo caricamento e PWA minima; risorse documentali non usate dalla UI corrente devono essere opzionali o rimosse dal precache.
5. Ogni `event.respondWith()` deve ricevere una Promise che si risolve sempre a una `Response`.

## Proposte per slice successive

### CML-529 - Fetch Fallback Always-Response Microfix

- Introdurre helper locale `offlineResponse(request)` o equivalente.
- Garantire fallback finale esplicito nei rami navigation e non-navigation.
- Evitare `index.html` come fallback generico per asset non HTML.
- Valutare gestione esplicita per cross-origin GET.

### CML-530 - Offline Test and Cache Update

- Testare install, navigazione offline, asset offline, cache vuota e storage cancellato.
- Aggiornare `CACHE_NAME` solo se la modifica runtime lo richiede.
- Verificare browser live post-deploy se la slice modifica `_published_snapshot/netlify-current/**`.
