## Contratto: punti chiave

### 1. Schema family

`.cml v1.0` è la famiglia schema supportata. Unico valore accettato: `"1.0"`.

### 2. Tipi di file accettati

- `teacher_proposal` — esportato da docente, importato da dipartimento
- `department_outcome` — esportato da dipartimento, importato da referente

Il referente **non esporta `.cml`**; esporta report Markdown.

### 3. Matrice ruolo/file-type

| Ruolo        | Esporta                     | Importa              | fileType accettato   |
| ------------ | --------------------------- | -------------------- | -------------------- |
| Docente      | `teacher_proposal` `.cml`   | Nessuno              | N/A                  |
| Dipartimento | `department_outcome` `.cml` | `teacher_proposal`   | `teacher_proposal`   |
| Referente    | Report Markdown             | `department_outcome` | `department_outcome` |

File con ruolo sbagliato sono bloccati al gate `fileType`.

### 4. Campi teacher_proposal

**Top-level required**: `schemaVersion`, `fileType`, `discipline`, `counts` (oggetto), `proposals` (array), `humanValidationRequired` (true).

**Top-level optional**: `appName`, `createdAt`, `sourceContext`.

**Top-level advisory**: `role`, `checks`.

**Proposal item required**: `id`, `discipline`, `proposta` (non-vuoto).

**Proposal item optional**: `ordine`, `classe`, `type`, `status`, `decisione`, `testoAttuale`, `motivazione`, `fonte`.

### 5. Campi department_outcome

**Top-level required**: `schemaVersion`, `fileType`, `discipline`, `disciplineWorkStatus`, `proposalHandling` (array), `checks` (oggetto), `humanValidationRequired` (true).

**Top-level optional**: `appName`, `createdAt`, `disciplines` (array).

**Top-level advisory**: `role`, `checks`.

**Handling item required**: `proposalId` (non-vuoto).

**Handling item optional**: `handling`, `note`.

**Accepted handling values**:

- `confluita_nella_sintesi`
- `riformulata_dal_dipartimento`
- `assorbita_in_altra_proposta`
- `da_chiarire`
- empty/null → `senza_esito`

### 6. Policy versioni

- **Missing `schemaVersion`**: accettato come legacy v1.0 con warning `legacy_version`.
- **Supported `"1.0"`**: accettato.
- **Unsupported future version**: bloccato con `unsupported_schema`.
- **Malformed value (non-string)**: bloccato come structurally malformed.

### 7. Compatibilità backward

Tutti i file `.cml` esportati da CurManLight con `schemaVersion: "1.0"` (o senza) sono backward-compatibili. Esempi correnti, file stored, e file di handoff sono valid input. Nessuna azione di migrazione richiesta.

### 8. Duplicate policy

- Duplicate file identity: fingerprint basato su `discipline`, `createdAt`, e contenuto proposte/handling.
- Duplicate proposal id: warning-only, proposta entra comunque nello stato con chip "Duplicato probabile".
- Duplicate department outcome id/handling id: warning-only per file; handling duplicati dentro un singolo file sono bloccati come strutturalmente invalidi.
- Stessa disciplina ripetuta in file separati: permessa.

### 9. Mixed-discipline policy

- **Permesso con warning**: batch import con discipline miste da docenti.
- **Bloccato**: `department_outcome` con `discipline` vuota e `disciplines` assente/vuota.
- **Single-discipline**: warning di mixed-discipline neutralizzato se tutte le proposte condividono la stessa disciplina.
- **Multi-discipline package**: non definito in v1.0; future estensioni devono preservare compatibilità v1.0.

### 10. discipline / disciplines consistency

- `discipline` è l'identificatore primario.
- `disciplines[]` è un riepilogo derivato.
- Se `disciplines.length === 1`, `discipline` deve essere `disciplines[0]`.
- Se `disciplines.length > 1`, `discipline` deve essere `disciplines[0]` (sorted).
- Divergenza: warning, non blocco.

### 11. Item-level validation expectations

**Proposal items**:

- `id`: warning se assente/vuoto
- `proposta`: warning se vuota
- `motivazione`: warning se vuota
- `fonte`: warning se vuota
- `discipline`: fallback a top-level, warning se assente nel singolo item

**Handling items**:

- `proposalId`: warning se assente/vuoto
- `handling`: warning se vuoto/null (diventa `senza_esito`)
- `note`: non validato, advisory

**Status/decision fields**: non enforced come enum; valori sconosciuti trattati come stringhe libere.

**Source/context**: `sourceContext` è advisory metadata, non validato.

### 12. Malformed-file e wrong-type handling

- `not_recognizable`: estensione non `.cml`, top-level array/null — bloccato.
- `invalid_json`: `JSON.parse` fallisce — bloccato.
- `wrong_type`: `fileType` non corrisponde all'importer — bloccato.
- `missing_data`: campi top-level required assenti — bloccato con elenco campi mancanti.
- Valori sconosciuti aggiuntivi: preservati, ignorati.

### 13. Import size e large-array policy

- **File size**: warning sopra 5 MB, blocco sopra 20 MB.
- **Array size**: warning sopra 500 items, blocco sopra 2,000 items.
- **String field size**: troncamento con warning sopra 100,000 caratteri per campo.

Motivo: ambiente browser senza backend; oversized import possono bloccare la UI o esaurire memoria.

### 14. Privacy/security guidance

- Nessun dato personale studente in `.cml`.
- Nessuna credenziale/secret in `.cml`.
- Drive endpoint da `localStorage` è untrusted boundary; nessun allowlist attuale.
- Fallback locale/manuale consigliato in produzione.
- Import locale senza rete; rendering con `escapeHtml()` prima di `innerHTML`.

### 15. Tassonomia errori utente

**Blocking errors**: `not_recognizable`, `invalid_json`, `wrong_type`, `missing_data`, `unsupported_schema`, `too_large`.

**Warnings**: `legacy_version`, `missing_semantic_field`, `discipline_mismatch`, `duplicate`, `mixed_discipline`.

**Informational**: successi, assenza endpoint Drive, nessun file valido.

**Recovery guidance**: ogni blocking/warning include azione suggerita (rigenera, verifica ruolo, split file, ecc.).

### 16. CML-200 implementation scope

CML-200 implementa:

1. Enforcement `schemaVersion === "1.0"` con block/warn.
2. Item-level structural checks con warning per `id`, `proposta`, `motivazione`, `fonte`, `proposalId`.
3. Duplicate detection con fingerprint strategy.
4. Size/array limits con warning/block.
5. `discipline`/`disciplines` consistency enforcement.
6. Messaggi utente con recovery guidance strutturata.
7. Preservazione campi sconosciuti.
8. `code` strutturato in tutti i risultati di validazione.

### 17. CML-201 smoke scope

CML-201 verifica end-to-end:

- Docente esporta `teacher_proposal` singola/multi-disciplina.
- Dipartimento importa batch misti, JSON malformato, tipo sbagliato, `schemaVersion` mancante/non supportato, duplicati, e esporta `department_outcome`.
- Referente importa esiti validi, handling vuoti (`senza_esito`), discipline mismatch, e esporta report Markdown.
- Tutti gli esempi `.cml` in `docs/04_user/esempi_cml/` passano come valid input senza modifiche.
