# CML-059 — GITHUB_PAGES_STATIC_DEPLOY_SELECTION_AND_SETUP

## Obiettivo
Selezionare e predisporre GitHub Pages come canale gratuito di pubblicazione statica per CurManLight, in alternativa a Netlify bloccato per crediti, usando la cartella runtime già validata `_published_snapshot/netlify-current`.

## Motivo scelta GitHub Pages
GitHub Pages offre hosting gratuito per siti statici direttamente da un repository GitHub, senza costi aggiuntivi e con integrazione nativa tramite GitHub Actions. È coerente con l'obiettivo di costo zero immediato e non richiede spostamento del runtime.

## Motivo scarto Coolify
Coolify richiede un'istanza già disponibile e configurata, il che comporta un costo operativo o di setup non immediato. Non è stato selezionato per mantenere la scelta puramente gratuita e senza dipendenze da infrastrutture esterne da provisioning.

## Motivo scarto Netlify production per ora
Il production deploy su Netlify è bloccato a causa di crediti esauriti (`Account credit usage exceeded`). Mentre il draft deploy è disponibile, non è possibile promuovere in produzione senza aggiungere crediti. Si opta per un'alternativa gratuita in attesa di risolvere la questione crediti su Netlify.

## Workflow adottato
Si è scelto di utilizzare un workflow GitHub Actions che pubblica la cartella `_published_snapshot/netlify-current` su GitHub Pages. Il workflow è definito in `.github/workflows/pages.yml` e si triggera su:
- `workflow_dispatch` (esecuzione manuale)
- `push` sulla branch `cml-008r-fix-markdown-decision-summary` limitato ai cambiamenti nella cartella di pubblicazione e nel workflow stesso.

Il workflow utilizza le azioni ufficiali:
- `actions/checkout@v4`
- `actions/configure-pages@v5`
- `actions/upload-pages-artifact@v3`
- `actions/deploy-pages@v4`

## Cartella pubblicata
La cartella di origine per il publish è `_published_snapshot/netlify-current`, che contiene il runtime statale già validato (HTML, CSS, JS, service worker, etc.). Nessun build command è necessario.

## Istruzioni GitHub UI
Dopo aver aggiunto e committato il workflow:
1. Aprire il repository su GitHub.
2. Andare in `Settings` → `Pages`.
3. In `Build and deployment`, impostare `Source: GitHub Actions`.
4. Salvare.
5. Pushare la branch con il workflow (se non già fatto).
6. Avviare il workflow manualmente da `Actions`, se non parte automaticamente.

## Controlli eseguiti
- Verificata l'esistenza di `_published_snapshot/netlify-current/index.html`.
- Verificata la creazione del file workflow.
- Controllato che nessun file dentro `_published_snapshot/netlify-current` sia stato modificato (tramite `git diff --name-only -- _published_snapshot/netlify-current`).
- Controllato che i soli file modificati siano quelli autorizzati.

## Rischi residui
- L'URL del repository GitHub deve essere corretto e accessibile in push. Attualmente il remote punta a `https://github.com/anton/CurManLight`; se diverso, è necessario aggiornare `git remote set-url origin <URL>`.
- Il workflow richiede che la branch `cml-008r-fix-markdown-decision-summary` esista sul remote e sia pushabile.
- È necessario abilitare GitHub Actions nel repository (di solito abilitato di default).
- Il deploy su GitHub Pages potrebbe essere soggetto a limiti di banda o di utilizzo gratuiti, ma per un sito statico leggero come CurManLight è improbabile un superamento.

## Verdetto
Il setup del workflow GitHub Pages è completo. La slice è pronta per il passo successivo: impostare la fonte su GitHub Actions nelle impostazioni del repository e eseguire il workflow.
