## Findings

### 1. Evidence marking (CML-215 implementation)

- **localStorage key**: `cml_evidenze_state`
- **Schema**: `{ "unitId_ev_N": "adottata" | "adattata" | "non_applicabile" }`
- **Default**: `"proposta"` (non stored, applied in `getEvidenceState()`)
- **Reset**: `localStorage.removeItem('cml_evidenze_state')` via `resetEvidenceStates()`
- **Evidence IDs**: Composite from `unit.id + '_ev_' + index` — e.g., `tec_pri_1_001_ev_0`
- **Scoping**: Per all disciplines (single key), no discipline prefix

### 6. Privacy risks

| Risk                             | Severity         | Mitigation                                                           |
| -------------------------------- | ---------------- | -------------------------------------------------------------------- |
| Evidence states in localStorage  | None             | Only discipline/unit/evidence IDs, no personal data                  |
| UDA draft in localStorage        | Low              | Draft = titolo, disciplina, evidence IDs, criteria — no student data |
| Teacher name in UDA author field | Low              | Only name from settings, no ID, no role                              |
| Student data in UDA              | None by contract | CML-214 §10: vietati identificativi studente, voti, dati sensibili   |
| localStorage persistence         | Low              | Same as existing CML-215; user controls localStorage                 |

**Assessment**: UDA draft poses LOW additional privacy risk, provided contract invariants are respected.

### 7. UDA generation feasibility

A UDA draft CAN be generated from:

- **Selected discipline**: via `mappaDisciplinaCorrente` or active discipline context
- **Selected unit**: via `getDisciplineEvidenceData()` returning unit array
- **Evidence text**: from `unit.evidenze[]` array
- **Evidence state**: from `cml_evidenze_state` localStorage
- **Objectives/traguardi**: from `unit.obiettivi[]`, `unit.traguardo`
- **Criteria**: from `unit.criteriValutazione[]`

Reuses existing data flows — no new data needed.

### 8. Export readiness

The app already has proven markdown export infrastructure (`downloadMarkdownString`, `copyMarkdownToClipboard`). A UDA markdown export would:

- Build a markdown string from selected unit data + evidence states
- Use existing `downloadBlob()` for file download
- Use existing `copyMarkdownToClipboard()` for clipboard
- Not require any `.cml` schema changes

## Audit questions

### Q1: Can a UDA draft be generated from existing data?

**Yes.** All fields from CML-214 §4 (titolo, disciplina, nuclei, evidenze, obiettivi, attività sintesi, criteri, durata) can be derived from existing curriculum data + evidence marking. No new curriculum data needed.

### Q2: What should remain editable by the teacher?

- Titolo UDA (selection or custom)
- Attività sintesi (teacher-written)
- Durata ore (teacher input)
- Evidence selection (which evidenze to include — defaults to adottate)
- Note docente (free text)

### Q3: What should remain read-only from curriculum data?

- Testo evidenze (immutable from curricolo)
- Traguardi, obiettivi, conoscenze, abilità, criteri (curriculum source)
- Fonte curricolare

### Q4: localStorage or generated on demand?

**Generated on demand** for first increment (CML-217). localStorage persistence deferred to later slice. This minimizes risk: no draft management UI, no draft listing/deletion, no localStorage data migration.

### Q5: First export format?

**Markdown copy/download** (Option A in CML-214, expanded). Markdown is:

- Already proven in codebase
- Readable in any text editor
- Copyable to Word/Google Docs
- No library dependency
- No `.cml` schema change

### Q6: What must be excluded to avoid premature complexity?

- Persistent UDA model in localStorage (defer)
- UDA listing/management UI (defer)
- `.cml` export/import for UDA (defer)
- Student data fields (contract prohibition)
- Grade fields (contract prohibition)
- Department validation workflow for UDA (defer)
- Multi-teacher collaboration (defer)

### Q7: Minimum acceptance criteria for CML-217?

1. UDA draft preview generated on demand for selected discipline/unit
2. Evidence selection: defaults to evidenze marked `adottata`
3. Markdown export: clipboard copy + download
4. No curriculum JSON changes
5. No `.cml` schema changes
6. No student data stored
7. localStorage evidence marking unchanged (`cml_evidenze_state`)
8. UDA generated on demand (no persistent model)
9. Mobile responsive
10. Validator 14/14 PASS
11. Shape test 14/14 PASS
12. Warning microcopy: "bozza non approvata", "da adattare alla classe"

## Option comparison

| Option                                   | Risk    | Complexity | Utility | Verdict             |
| ---------------------------------------- | ------- | ---------- | ------- | ------------------- |
| **A** — Preview only, on demand          | Minimal | Low        | Medium  | ✅ Fallback         |
| **B** — Preview + markdown copy/download | Low     | Low        | High    | ✅ **Recommended**  |
| **C** — With localStorage persistence    | Medium  | Medium     | Medium  | ❌ Defer            |
| **D** — .cml export extension            | High    | High       | Low     | ❌ Reject           |
| **E** — Defer UDA, harden evidence       | Minimal | None       | Low     | ❌ Too conservative |

## Recommendation

**Select Option B: UDA draft preview + markdown copy/download**

Motivazione:

- Riusa pattern già testati (downloadBlob, copyMarkdownToClipboard, modelToMarkdown)
- Nessuna modifica schema `.cml`
- Nessun dato studente
- Nessun voto
- Nessun modello UDA persistente (generato on demand)
- localStorage solo per evidence marking (invariato)
- Allineato con contratto CML-214 §14
- Utility alta per docenti: può copiare/incollare UDA in documento condiviso
