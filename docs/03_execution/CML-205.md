## Obiettivo

Implementare hardening minimal della sicurezza per l'endpoint di upload Drive, seguendo il security boundary documentato in CML-199 e differito in CML-200.

## Root Cause / Context

La funzione `uploadTeacherCmlToDrive()` legge l'endpoint da `localStorage.getItem('curmanlight.driveUploadEndpoint')` e lo usa direttamente in `fetch()` senza alcuna validazione. Questo era il rischio P2 documentato in CML-198 e CML-199: "Optional Drive endpoint lacks documented trust boundary".

## Allowlist Policy Implemented

### Host Consentiti (espliciti)

```javascript
const ALLOWED_ENDPOINT_HOSTS = ['drive.google.com', 'docs.google.com']
```

### Regole di Validazione

| Check          | Regola                                                                       | Esito                                                                      |
| -------------- | ---------------------------------------------------------------------------- | -------------------------------------------------------------------------- |
| Empty          | `endpoint` vuoto o soli whitespace                                           | `empty` → fallback                                                         |
| Protocol       | Solo `https://` consentito                                                   | `http://`, `javascript:`, `data:`, `file:` → `https`/`protocol` → fallback |
| Malformed      | URL parse error                                                              | `malformed` → fallback                                                     |
| Credenziali    | `username:password@` nell'URL                                                | `credentials` → fallback                                                   |
| Localhost      | `localhost`, `127.x.x.x`, `192.168.x.x`, `10.x.x.x`, `172.16-31.x.x`, raw IP | `localhost` → fallback                                                     |
| Host allowlist | Solo host espliciti consentiti (con suffisso subdomain)                      | `host` → fallback                                                          |

### Subdomain Matching

Un host è accettato se:

- È esattamente nella lista `ALLOWED_ENDPOINT_HOSTS`
- È un subdomain diretto di un host consentito (es. `sub.school.edu` se `school.edu` è consentito)

## Comportamento di Fallback

| Scenario                               | Comportamento                                            |
| -------------------------------------- | -------------------------------------------------------- |
| Nessun endpoint configurato            | Messaggio informativo + download manuale `.cml`          |
| Endpoint non valido (qualsiasi motivo) | Messaggio di warning specifico + download manuale `.cml` |

Il download manuale `.cml` è sempre disponibile come fallback.

## Limitazioni Documentate

1. **Client-side only**: l'allowlist è un hardening lato client, non una sicurezza server-side. Un endpoint man-in-the-middle o un server compromesso possono bypassarla.
2. **Allowlist statica**: `ALLOWED_ENDPOINT_HOSTS` è hardcoded nel runtime. Per aggiungere domini scuola, è necessario modificare il sorgente e ridistribuire.
3. **Nessuna verifica TLS**: il codice non verifica certificati TLS oltre lo schema `https://`. La validazione si basa sull'URL dichiarato, non sulla risposta del server.
4. **Nessuna credenziale storage**: l'endpoint resta in `localStorage` senza cifratura. Non contiene credenziali proprie, ma un URL valido è un bersaglio per attacchi XSS se il browser è compromesso.
5. **Fallback manuale**: se l'allowlist rifiuta l'endpoint, l'utente deve caricare manualmente il `.cml`. Questo è il comportamento raccomandato in produzione.
