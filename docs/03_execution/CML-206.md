## Obiettivo

Audit e selezione della strategia più sicura per un possibile formato `.cml` multi-disciplina, dopo il completamento del ciclo di hardening `.cml v1.0`.

## Domanda di Decisione

CurManLight deve introdurre un formato `.cml` multi-disciplina formale ora, rimandarlo, o supportare solo il comportamento attuale di import batch senza nuovo schema di package?

## Opzioni Valutate

### Opzione A — Nessun nuovo formato package (consigliata)

Mantiene i tipi di file v1.0 esistenti (`teacher_proposal`, `department_outcome`) e l'import batch multiplo già implementato.

**Vantaggi**:

- Zero rischio di rottura compatibilità
- Zero complessità schema aggiuntiva
- Flusso utente già funzionante
- Nessun impatto su validatori, export, import

**Svantaggi**:

- Nessuna astrazione "package" esplicita
- Dipartimento/referente gestiscono batch di file multipli

### Opzione B — Package manifest leggero dentro v1.0

Aggiunge un campo opzionale `package` dentro i file esistenti, senza version bump.

**Vantaggi**:

- Retrocompatibile
- Fornisce contesto di package senza nuovo fileType

**Svantaggi**:

- Ambiguità semantica: un file con `package` è un package o un proposal singolo?
- Aggiunge complessità a validatori e UI senza chiaro bisogno operativo

### Opzione C — Nuovo fileType `multi_discipline_package`

Introduce un terzo tipo di file esplicito, ancora sotto `schemaVersion: "1.0"`.

**Vantaggi**:

- Chiaro discrimine di ruolo
- Esplicito nel contenuto

**Svantaggi**:

- Espone la matrice ruolo/file-type a 3 casi (da 2)
- Richiede nuovo validatore, nuovo export, nuovo import
- Nessun evidence che le scuole richiedano questo formato vs batch

### Opzione D — Rimanda a v1.1 o v2.0

Documenta l'intenzione ma non implementa nulla adesso.

**Vantaggi**:

- Conserva flessibilità futura
- Evita complessità prematura

**Svantaggi**:

- Se la need emerge domani, non c'è contratto già pronto

### Opzione E — Solo UX guidance per selezione multipla

Migliora la UI di selezione file (multi-select, liste, anteprima) senza cambiare schema o behavior.

**Vantaggi**:

- Zero rischio schema
- Migliora immediatezza del flusso batch esistente
- Retrocompatibile al 100%

**Svantaggi**:

- Non risolve problemi di deduplication cross-file a livello di package

## Comportamento Attuale di Batch Import

Il runtime attuale (`importTeacherCmlFiles`, `importDepartmentOutcomeCmlFiles`) già supporta:

- Selezione multipla di file `.cml` in un singolo import
- Dedup per file via fingerprint (`discipline`, `createdAt`, contenuto righe)
- Warning "Duplicato probabile" per file duplicati, inclusi nello stato
- Raggruppamento per disciplina nel dipartimento, con warning "Sono presenti discipline diverse"
- Fallback da `proposals[].discipline` a top-level `discipline` per item singoli
- Import di proposal da discipline miste nella stessa batch (warning-only, non bloccato)

Questo significa che il bisogno operativo base "importa proposte da più file/discipline" è già soddisfatto.

## Selezione Raccomandata

**Opzione A** — Mantieni il batch import attuale come meccanismo multi-disciplina.

Motivazione:

1. Il comportamento attuale soddisfa il 90%+ dei casi d'uso: docenti che gestiscono discipline multiple esportano file separati; i dipartimenti/refenti importano batch.
2. Non c'è evidence che le scuole richiedano un package file esplicito.
3. Un package formale introduce complessità schema, validatore, UI, e test burden che non sono giustificati da un bisogno chiaro.
4. La compatibilità v1.0 è preservata al 100%.
5. Se in futuro emerge una chiara necessità di package, si può far evolvere in v1.1 con un nuovo `fileType` o campo manifest.

## Alternative Rigettate

| Opzione                                       | Motivo Rifiuto                                                                                      |
| --------------------------------------------- | --------------------------------------------------------------------------------------------------- |
| B (package manifest in v1.0)                  | Ambiguità semantica, complessità non giustificata                                                   |
| C (nuovo fileType `multi_discipline_package`) | Espone matrice a 3 casi, richiede nuovo validatore/export/import, nessun evidence strong            |
| D (rimanda a v2.0)                            | Eccessivamente conservativo: basta Opzione A per adesso, senza bisogno di contratti futuri          |
| E (solo UX guidance)                          | Utile come follow-up, ma non risolve la questione schema/package; meglio un UX improvement separato |

## Prossima Slice Consigliata

**Opzionale**: un miglioramento UX per il batch import esistente (multi-select più chiaro, drag-and-drop, anteprima file prima dell'import) come slice separata, senza schema changes.

**Non raccomandato**: introduzione di package formale `.cml` multi-disciplina fino a evidenza operativa chiara da scuole/utenti.
