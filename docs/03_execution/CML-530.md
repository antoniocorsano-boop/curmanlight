- Base: `02207f555ad73848e8f50bb8a3768809d797bf23`

## Evidenze consolidate

CML-529 è pubblicata e verificata tramite GitHub Pages run `29559635946`, conclusa con esito `success`.

Il service worker pubblicato garantisce una `Response` esplicita in ogni fallback:

- navigazione offline senza rete e senza cache: HTTP `503`;
- richiesta non-navigation/asset senza rete e senza cache: HTTP `504`.

La verifica live è stata eseguita con profilo temporaneo, service worker e Cache Storage puliti. La pagina online ha risposto HTTP 200, il service worker si è attivato e i due casi offline hanno restituito gli status attesi senza page error.

## Contratto offline

1. La prima visita online installa e attiva il service worker e tenta il precache delle risorse `APP_SHELL`.
2. Una risorsa non recuperabile durante il precache non invalida globalmente l'installazione.
3. Le navigazioni privilegiano la rete; in assenza di rete usano la cache disponibile.
4. Se nessuna risposta di rete o cache è disponibile, una navigazione restituisce 503.
5. Le richieste non-navigation usano cache/rete secondo la strategia corrente.
6. Se nessuna risposta è disponibile, una richiesta non-navigation restituisce 504.
7. Nessun ramo `event.respondWith()` può risolversi a `undefined`.
8. Gli status 503 e 504 sono risposte tecniche controllate, non equivalgono a disponibilità offline completa dell'applicazione.

## Matrice minima di verifica

- online, cache vuota: pagina HTTP 200 e service worker attivo;
- offline, cache popolata: navigazione servita dalla cache quando disponibile;
- offline, cache vuota, navigation: Response 503;
- offline, cache vuota, asset: Response 504;
- seconda ricarica dopo installazione: nessun errore service worker;
- console/page errors: nessun errore inatteso.

## Esclusioni intenzionali

- nessuna nuova pagina offline;
- nessuna modifica ad `APP_SHELL`;
- nessuna modifica al percorso motto;
- nessuna modifica a HTML, React, dati o workflow;
- nessuna promessa di funzionamento offline completo.
