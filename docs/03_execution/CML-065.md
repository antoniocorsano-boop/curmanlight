# CML-065 — Product Model and Repository Alignment Contract

## Obiettivo

Definire una volta per tutte il modello concettuale di CurManLight (Curriculum vs Didattica) e allineare il repository. Docs-only, nessuna modifica runtime.

| Evento                                                     | Riferimento                  |

| CML-063A raccomandava runtime Home                         | Sospeso: prima serve modello |
| Tecnologia normalizzata rischiava di diventare quarta area | CML-062/064/064A             |
| Contratto dati normalizzato già definito                   | CML-061                      |
| Codice operativo già implementato                          | CML-055/056                  |

## Perché CML-063A (runtime Home) è stato sospeso

Prima di aggiungere un cruscotto Home, bisogna sapere **quali aree** deve mostrare. Aggiungere una Home senza modello di prodotto significa creare un'altra card per ogni nuova funzione, tornando al problema dei troppi pulsanti. Questo contratto fissa il modello, poi la Home (CML-065A) lo implementa.

## Modello scelto

| Area           | Definizione                                                                  | Stato                              |
| -------------- | ---------------------------------------------------------------------------- | ---------------------------------- |
| **Curriculum** | Documento istituzionale: consultazione, revisione, validazione, esportazione | Attivo (tutte le funzioni attuali) |
| **Didattica**  | Progettazione operativa: UDA, attività, materiali, valutazione               | Futuro (vuota, da popolare)        |

### Posizione di Tecnologia normalizzata

**Curriculum > Consultazione** — pacchetto disciplinare pilota, non area, non card, non tab.

### Struttura target Home

2 card: Curriculum (4 link secondari) + Didattica (4 link secondari). Niente azioni operative in Home.
