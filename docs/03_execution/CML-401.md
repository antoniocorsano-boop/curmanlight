# CML-401 — Process ecosystem Home runtime remote lab

## Stato

- Branch laboratorio remoto: `remote-lab/cml-401-process-home-runtime-keep`
- Base: `codex/cml-400-process-ecosystem-home-structure`
- Tipo: remote-lab mantenibile
- Main modificato: NO
- Deploy: NO

## Obiettivo

Preparare una integrazione runtime simulando un lavoro locale, ma conservando tutto su un ramo remoto stabile e mantenibile.

La Home deve diventare un hub del processo curricolare completo, rendendo visibili:

- fonti ufficiali;
- consultazione;
- revisione;
- progettazione;
- esportazioni;
- processo Docente → Dipartimento → Referente → Collegio;
- file di lavoro CurManLight `.cml`;
- validazione umana obbligatoria;
- dati locali e nessun invio automatico.

## Decisione tecnica

Il file runtime `index.html` è un all-in-one esteso. Da connettore remoto la sostituzione completa del file è tecnicamente possibile ma rischiosa, perché non equivale a una patch locale controllata.

Per simulare il locale senza compromettere il runtime, questa branch contiene:

1. un frammento HTML/CSS pronto per la Home;
2. uno script applicativo che modifica `index.html` e `_published_snapshot/netlify-current/index.html` se eseguito in un clone locale o in un ambiente remoto con checkout completo;
3. una traccia di verifica e criteri di accettazione;
4. nessuna modifica diretta a `main`.

## Strategia di patch

Lo script applicativo opera in modo prudente:

- controlla che entrambi i file runtime esistano;
- controlla che root e snapshot contengano i marcatori attesi;
- inserisce CSS dedicato prima della chiusura `</style>`;
- inserisce un blocco Home prima delle card operative esistenti, usando un marcatore CML-401;
- evita duplicazioni se il blocco è già presente;
- mantiene root e snapshot allineati;
- non modifica dati curricolari;
- non modifica schema `.cml`;
- non introduce nuove dipendenze.

## Criteri di accettazione runtime

La futura esecuzione della patch dovrà produrre:

- Home con sezione `Hub del processo curricolare`;
- percorso `Consulta → Scegli → Progetta → Esporta → Verifica`;
- pipeline `Docente → Dipartimento → Referente → Collegio`;
- card aree sistema: Curricolo, Progetta, Esportazioni, Guida;
- governance: dati locali, nessun invio automatico, validazione umana;
- root e snapshot entrambi aggiornati;
- nessun cambiamento ai dati curricolari.
