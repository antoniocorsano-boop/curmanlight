## Summary Verdict

The current workflow is functional but contract-light. It uses two JSON-backed `.cml` file types:

- `teacher_proposal`
- `department_outcome`

Both use `schemaVersion: "1.0"`, but versioning is not enforced by imports and there is no central schema contract or compatibility policy. Import handling is intentionally local, no-network, and reasonably safe for malformed files, wrong file types, and XSS rendering. The main risks are schema drift, weak semantic validation, duplicate handling that warns but does not suppress, and user-facing messages that hide exact remediation.

## Minimal CML-199 Scope

CML-199 should remain a contract slice before implementation:

1. Define a formal `.cml` contract for `teacher_proposal` and `department_outcome` version `1.0`.
2. Classify required, optional, derived, and deprecated fields.
3. Specify version acceptance rules for missing/unknown/unsupported `schemaVersion`.
4. Define compatibility rules for current examples and current runtime exports.
5. Define error taxonomy and user-facing message requirements without changing runtime behavior yet.
6. Define duplicate policy as warn-only, ignore, or merge candidate.
7. Define role/file-type matrix and wrong-role behavior.
8. Identify CML-200 validation implementation scope and CML-201 smoke matrix.
