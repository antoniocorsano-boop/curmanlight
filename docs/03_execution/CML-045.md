### Strategie

| Evento     | Comportamento                                                                                                                                                                         |
| ---------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| `install`  | Apre cache con `CACHE_NAME`, precarica `APP_SHELL` tramite `cache.addAll()`. Chiama `self.skipWaiting()`.                                                                             |
| `activate` | Elimina tutte le cache con nome diverso da `CACHE_NAME`. Chiama `self.clients.claim()`.                                                                                               |
| `fetch`    | **Cache-first** per tutte le richieste GET: (1) cerca in cache, (2) se trovato → restituisce, (3) se non trovato → fetch + cache (solo stesso origin), (4) fallback a `./index.html`. |

### Criticità rilevate

- **`skipWaiting()`** ✅ presente — il nuovo SW prende controllo immediato all'installazione
- **`clients.claim()`** ✅ presente — il nuovo SW controlla tutti i client aperti
- **Pulizia vecchie cache** ✅ presente — `activate` elimina cache con nome diverso
- **`index.html` in APP_SHELL** ⚠️ precaricato all'install → mai ricalcolato finché il nome cache non cambia

## Rischio cache

| Fattore                 | Valore                                                                                                                                                                                                                            |
| ----------------------- | --------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **Classificazione**     | **MEDIO**                                                                                                                                                                                                                         |
| **Causa**               | Cache name invariato da `cd5996e`; 26 commit di modifiche a `index.html` (viewer curricolo, etichette versione, callout sezioni generali, fix flusso dipartimento `.cml`, report gruppo curricolo) non accompagnati da bump cache |
| **Impatto utente**      | Utenti già presenti prima degli aggiornamenti vedono versione obsoleta (senza viewer, senza etichette, senza flusso dipartimento completo). Nuovi visitatori o chi cancella la cache vede la versione corretta                    |
| **Probabilità**         | MEDIA — colpisce solo utenti di ritorno con cache popolata prima delle ultime deploy; utenti nuovi e navigazione in incognito non coinvolti                                                                                       |
| **Rimedio consigliato** | Bump cache name in `sw.js` (una riga) in CML-046. `skipWaiting()` + `clients.claim()` già presenti → effetto immediato                                                                                                            |

### Perché non ALTO

- `skipWaiting()` e `clients.claim()` già implementati correttamente
- Il problema si risolve con un deploy successivo che modifichi `sw.js`
- La cache non blocca l'accesso — solo la versione visualizzata
- Nessun dato utente coinvolto

### Perché non BASSO

- 26 commit di differenza tra ultimo bump cache e HEAD
- Cache-first strategy senza riconvalida periodica
- L'utente non ha modo di forzare l'aggiornamento senza azioni manuali (cancellazione dati browser)

## Decisione

Motivazione:

- Intervento minimo: 1 carattere (cifra del suffisso cache)
- `skipWaiting()` e `clients.claim()` già presenti → nessuna altra modifica a `sw.js`
- L'activate handler elimina automaticamente le vecchie cache
- Nessun cambiamento alla strategia offline
- Nessun cambiamento alla strategia fetch
- Nessun rischio di regressione
