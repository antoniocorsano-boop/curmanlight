## Analisi delle alternative

### 1. Ulteriore maturità UX

**Descrizione:** Continuare con interventi runtime sui PM incompleti (PM-03 al 45%, PM-06 al 70%, PM-07 al 40%).

**Benefici:**
- Miglioramento continuo dell'esperienza utente
- Avanzamento verso obiettivi di maturità definiti
- Interventi mirati su problemi specifici

**Rischi:**
- Rischio di interventi cosmetici non necessari
- Possibile sovrapposizione con validazione utente
- Difficoltà a identificare priorità senza feedback reale

**Fattibilità:** Alta (pattern consolidato)

### 2. Documentazione d'uso

**Descrizione:** Creare guida utente sintetica con istruzioni per docente, dipartimento, referente e percorso operativo essenziale.

**Benefici:**
- Supporto all'adozione del sistema
- Riduzione della curva di apprendimento
- Base per validazione con utenti

**Rischi:**
- Documentazione potrebbe diventare obsoleta rapidamente
- Rischio di duplicare contenuti già presenti in Guida e onboarding
- Tempo di creazione significativo

**Fattibilità:** Media (richiede sintesi efficace)

### 3. Validazione con utenti

**Descrizione:** Predisporre protocollo di test leggero con micro-scenario per docente e raccolta evidenze qualitative.

**Benefici:**
- Feedback reale su usabilità e comprensione del percorso
- Identificazione di problemi non evidenti internamente
- Base empirica per decisioni future
- Validazione dell'investimento fatto nella tranche PM-03/PM-06/PM-07

**Rischi:**
- Richiede coordinamento con utenti reali
- Tempo di esecuzione variabile
- Possibile identificazione di problemi critici che richiedono refactoring

**Fattibilità:** Media (richiede disponibilità utenti)

### 4. Chiusura milestone

**Descrizione:** Marcare uno stato pubblicabile, creare riepilogo di release e definire blocco stabile successivo.

**Benefici:**
- Formalizzazione del lavoro completato
- Base per comunicazione esterna
- Punto di riferimento per rollback

**Rischi:**
- Potrebbe essere prematuro senza validazione utente
- Rischio di creare documentazione di release che diventa obsoleta
- Possibile percezione di "chiusura" quando il sistema è ancora in evoluzione

**Fattibilità:** Alta (processo documentale)

## Scelta raccomandata

### Validazione con utenti

**Motivazione:**

Dopo molte tranche interne di maturità (CML-371 → CML-396), il sistema è abbastanza stabile per essere verificato con scenari reali. Non serve ancora aggiungere funzioni: serve capire se docente, dipartimento e referente comprendono davvero il percorso **consulta → scegli → progetta → esporta → verifica**.

**Razionalità:**

1. **Sistema stabile:** Tutte le validazioni runtime sono superate, il deploy è funzionante, il lessico è consolidato.

2. **Feedback reale necessario:** Dopo 14 slice di interventi interni, è fondamentale verificare che le modifiche siano effettivamente utili per gli utenti finali.

3. **Base empirica:** La validazione con utenti fornirà dati concreti per decidere se continuare con ulteriori interventi UX o pivotare verso altre direzioni.

4. **Rischio contenuto:** Il protocollo di test può essere leggero (micro-scenario) e non richiede cambiamenti al sistema.

5. **Allineamento con obiettivi PM:** PM-09 (Validazione con utenti) è attualmente allo 0% e rappresenta un'area di maturità non ancora esplorata.
