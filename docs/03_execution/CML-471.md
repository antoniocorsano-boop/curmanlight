# CML-471 — Field-Level Gap Contract Consolidation

## Obiettivo

Consolidare il contratto field-level introdotto con CML-470, separando la semantica della decisione dal solo campo legacy `testoFinale`.

- Branch base: `main`
- Commit iniziale: `4d250aa445875f3a2350197dc9b32c5ff3e4f64b`
- Baseline: merge CML-470, primo pilot Gap Layer field-level per Educazione Fisica.

## Branch

`codex/cml-471-field-level-gap-contract-consolidation`

## Problema rilevato

Il flusso B03 conosceva già `targetField`, ma la decisione operativa conservava principalmente una rappresentazione testuale completa in `testoFinale`. Questa forma non distingueva in modo strutturale:

- valore vigente del campo;
- valore proposto;
- valore effettivamente deciso;
- fotografia completa dell’unità risultante.

Il limite era particolarmente rilevante per i campi a lista (`obiettivi`, `conoscenze`, `abilita`, `evidenze`, `criteriValutazione`).

## Contratto consolidato

È stato introdotto `CurriculumFieldValue`:

```ts
type CurriculumFieldValue = string | string[]
```

I campi supportati restano l’insieme chiuso già definito da `GapTargetField`:

- `traguardo`;
- `obiettivi`;
- `conoscenze`;
- `abilita`;
- `evidenze`;
- `criteriValutazione`.

La nuova struttura `FieldDecisionSnapshot` conserva:

- `targetField`;
- `valoreVigente`;
- `valoreProposto`;
- `valoreDeciso`;
- `fotografiaUnita`.

## Funzioni di dominio

Il file `curman-react/src/lib/field-decision.ts` introduce funzioni pure per:

- `getCurrentFieldValue`;
- `normalizeFieldValue`;
- `normalizeProposedFieldValue`;
- `applyFieldDecisionToUnit`;
- `composeCandidateUnitSnapshot`;
- `serializeFieldValue`.

`applyFieldDecisionToUnit` crea una nuova unità e modifica esclusivamente il campo bersaglio. L’unità curricolare originale non viene mutata.

## Integrazione B03

`DecisioneActions` ora:

1. legge il valore vigente con il tipo reale;
2. normalizza il valore proposto;
3. determina il valore deciso in base all’azione;
4. compone una fotografia completa dell’unità;
5. registra il nuovo `fieldDecision` nello store;
6. deriva `testoFinale` solo per compatibilità con il modello legacy.

Le azioni coperte sono:

- accogli proposta;
- mantieni vigente;
- personalizza;
- chiedi revisione;
- riapri.

Per una richiesta di revisione non viene inventato un valore deciso o una fotografia candidata.

## Ruolo di `testoFinale`

`testoFinale` resta temporaneamente presente per:

- compatibilità con `Decisione` legacy;
- esportazioni o chiamanti non ancora migrati;
- rappresentazione testuale completa già esistente.

Non è più l’unica fonte semantica della decisione field-level.

## Persistenza

Il payload passa da:

```text
cml-work-decisions-v1
```

a:

```text
cml-work-decisions-v2
```

La chiave locale resta invariata per consentire il recupero dei dati precedenti.

Il parser accetta:

- `cml-work-decisions-v2`;
- `cml-work-decisions-v1`;
- `cml-local-v3`.

Le decisioni `v1` prive di `fieldDecision` vengono mantenute leggibili con `fieldDecision: null`. La riapertura gestisce questo caso senza generare una fotografia inventata.
