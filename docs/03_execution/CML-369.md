# CML-369 — External Provider Readiness and Discipline Data Access Audit

## Mandatory Slice Header
- Macroprogramma: (autonomo — docs-only audit)
- Dipende da: CML-368 (PR/publish chiusura)
- Tipo slice: docs-only audit
- Perimetro: nessuna modifica a runtime, curriculum, tools, assets, manifest, service-worker

## Obiettivo
Audit dei pattern di accesso ai dati disciplinari nel runtime e valutazione della readiness per provider esterni (Drive, AI, Second Brain/Obsidian). Identificare rischi, duplicazioni e gap contrattuali prima di implementare nuove integrazioni.

## Azioni

### Ricognizione

1. **Accesso dati disciplinari**: censiti tutti i pattern `ALL_CURRICULUM_DATA`, `TECNOLOGIA_NORM_DATA` e `discKey === 'tecnologia'` in `index.html`
2. **Hook Drive**: analizzate `uploadTeacherCmlToDrive()`, `getDriveUploadEndpoint()`, `validateDriveEndpoint()`, `isAllowedHost()` e whitelist `ALLOWED_ENDPOINT_HOSTS`
3. **Funzioni export**: censite 13 funzioni di export (Markdown, .cml, DOCX, PDF, backup, clipboard)
4. **Modello dati docente**: analizzato `buildTeacherCmlModel()` e `exportTeacherCml()`
5. **Security posture**: verificata validazione endpoint, anti-injection, host whitelist, fallback UX

## Rilievi

### R1 — Accesso dati Tecnologia (P1)
- L'eccezione `discKey === 'tecnologia'` è presente in 3 funzioni: `getDisciplineEvidenceData()`, `getAnnualDraftUnitsFor()`, `renderTecnologiaNorm()`
- `TECNOLOGIA_NORM_DATA` è un alias di `TECNOLOGIA_NORM`, costante inline con 16 unità (~linea 6202)
- Tecnologia è presente ANCHE dentro `ALL_CURRICULUM_DATA.tecnologia` — duplicazione fragile e costo byte inutile
- **Non correggere in CML-369.** Raccomandata slice successiva dedicata per centralizzare l'accesso dati

### R2 — Hook Drive (P2)
- Hook tecnico presente e difensivo: HTTPS-only, anti-injection, host whitelist (`drive.google.com`, `docs.google.com`), blocco IP privati
- Manca UI di configurazione: endpoint impostabile solo via `localStorage` da dev tools
- Non pronto per uso scolastico generalizzato
- **Non implementare UI ora.** Da rivalutare dopo contratto provider

### R3 — Provider esterni (P2/P3)
- Esistono export e hook che possono diventare provider (Drive, AI)
- Manca un contratto comune (interfaccia astratta, configurazione, error handling)
- Raccomandato contratto provider prima di nuove integrazioni

### R4 — Second Brain / Obsidian (candidato più sicuro)
- Export-only Markdown: nessun endpoint, nessuna chiave, nessun invio dati
- Pronto per documentazione utente; nessun rischio sicurezza

### R5 — AI provider (HOLD)
- Da non implementare prima di policy istituzionale e modello provider
- API key in `localStorage` non adatta a dispositivi condivisi scolastici

## Raccomandazione prossima slice
**CML-370 — Discipline Data Access Contract and Centralization Plan**

Motivazione: prima di introdurre provider esterni o export evoluti, serve definire e poi centralizzare l'accesso ai dati disciplinari, eliminando la fragilità dell'eccezione Tecnologia duplicata.
